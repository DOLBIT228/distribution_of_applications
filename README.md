# Розподіл заявок Bitrix24 (Streamlit)

Це Streamlit-скрипт для рівномірного розподілу заявок між менеджерами у вказаному статусі конкретної воронки Bitrix24.

## Що робить

- Вхід в локальний інтерфейс (логін/пароль з `st.secrets`).
- Логін у CRM не потрібен: зв'язок з Bitrix24 працює через ваш webhook.
- Вибір напрямку (воронка + статус + наступний статус).
- Показ кількості доступних угод у статусі.
- Вибір менеджерів для розподілу.
- Підтримка паралельної роботи декількох менеджерів з вибором активної зміни.
- Розподіл пачками (за замовчуванням `3` на одного вільного менеджера за запуск).
- Кнопка `Почати авто-розподіл`: запускає автоматичний цикл, який сам перевіряє воронку і розподіляє заявки протягом дня, поки не натиснути `Зупинити авто-розподіл`.
- Таблиці та статуси в інтерфейсі автоматично оновлюються в авто-режимі.
- Сповіщення в чат-бот:
  - коли авто-розподіл стартує;
  - коли авто-розподіл ставлять на паузу (з обов'язковою причиною);
  - коли авто-розподіл зупиняють (з обов'язковою причиною + звіт по менеджерах/типах заявок).
- Перед запуском перевіряє зайнятість менеджера у статусі `in_progress_status_id` ("Угода в роботі"):
  - якщо у менеджера є хоча б 1 активна угода, нова пачка не видається;
  - коли у менеджера 0 активних угод, він знову потрапляє у видачу пачки.
- Автоматичне оновлення відповідального (`ASSIGNED_BY_ID`) **і** переведення угоди в `next_status_id`.
- Логіка розподілу за напрямком:
  - **site** (також використовується для поточного напрямку `ТЕСТ`):
    - класифікація заявок за джерелом у типи `Лендинг` / `Сайт`,
    - балансування з урахуванням типу заявки.
  - **instagram**:
    - усі заявки розподіляються як тип `Інстаграм`,
    - балансування за загальною кількістю заявок між менеджерами.
- Денна таблиця розподілу (по менеджерах і типах) зберігається у локальній SQLite базі `distribution_history.db`.

## Налаштування

1. Встановіть залежності:

```bash
pip install -r requirements.txt
```

2. Створіть файл `.streamlit/secrets.toml` на базі `.streamlit/secrets.toml.example`.

3. Заповніть:
   - `bitrix.webhook_url` — webhook для API Bitrix24,
   - `[[auth.users]]` — користувачі для входу в інтерфейс (login/password/manager_id),
   - `[[directions]]` — напрямки:
     - `name` — назва,
     - `funnel_id` — ID воронки,
     - `status_id` — поточний статус, з якого беруться заявки,
     - `next_status_id` — наступний статус, куди переводяться заявки після розподілу,
     - `in_progress_status_id` — статус "Угода в роботі" для перевірки, що менеджер звільнив пачку,
     - `distribution_logic` — `site` або `instagram` (необов'язково, за замовчуванням визначається з назви напрямку),
     - `batch_size` — розмір пачки на одного вільного менеджера (необов'язково, за замовчуванням `3`).
     - `auto_interval_seconds` — інтервал авто-перевірки в секундах (необов'язково, за замовчуванням `30`).
   - `[[managers]]` — менеджери для розподілу (name + id).
   - `chatbot` — параметри бота (один з варіантів):
     - `webhook_url` — URL вашого вхідного webhook (payload: `{"text": "..."}`), або
     - `telegram_bot_token` + `telegram_chat_id` — для прямої відправки в Telegram Bot API.

## Запуск

```bash
streamlit run app.py
```

## Важливо

- Для `status_id` / `next_status_id` / `in_progress_status_id` використовуйте технічні значення стадій (наприклад `NEW`, `C2:NEW`).
- Скрипт працює тільки з тими воронками/стадіями, які ви вкажете в `secrets.toml`.
- Денна таблиця зберігається локально у файлі `distribution_history.db`.
